# SPARC AutoTutor Current AI Prompt Table

## Scope

This document inventories the current learner-facing AI utterance prompts used by the SPARC AutoTutor dialogue runtime. It covers only:

- the seven active SPARC AutoTutor move definitions in `learning-components/units/sparcsession/sparcMoveDefinitions.ts`; and
- the shared SPARC utterance prompt wrapper in `mofacts/client/views/experiment/svelte/services/sparcControllerDialogueOpenRouter.ts`.

The non-SPARC AutoTutor session and its prompt implementation are outside the scope of this table.

## Implementation Status

The `New prompt policy` column was implemented in the runtime on 2026-07-13. The authoritative runtime source is `learning-components/units/sparcsession/sparcMoveDefinitions.ts`; this table remains an audited design snapshot. The runtime misconception branches also carry the plan's safety requirement not to characterize repetition or endorsement of an active misconception as progress, closeness, or a good start.

The SPARC scoring envelope no longer carries a scorer-generated `evidence` string or an `addressed` flag. Score rows are numeric learner-model updates; omitted targets retain their durable prior values. The utterance model receives the learner's actual response and current merged planner state rather than a scorer-authored paraphrase.

The trial-display bridge now requires the authored `opening-tutor-message.value` problem statement and passes that same text directly to both the scoring request and the utterance request. Missing or blank problem text is a boundary error; the runtime does not substitute other display content.

Completion now has one terminal contract: graduation includes both expectation coverage and the authored active-misconception allowance, while the total learner-turn limit always selects `summary`. Only that terminal summary locks the dialogue, after which the SPARC surface presents a Continue button. Authored `dialogue.graduation.maxTurns` remains authoritative; when omitted, the runtime default is 50 learner turns.

Learner questions now bypass ordinary scaffold escalation. A legitimate question about the current content selects `question-deferral`; an off-topic or inappropriate question selects `question-scope-refusal`. Both moves preserve the current scaffold stage so the next content-focused response resumes the same forward chain. Expectation coverage remains the highest coverage demonstrated anywhere in the dialogue; later shorthand, omission, or context-dependent restatement does not lower it.

## Shared SPARC Utterance Prompt

The shared system prompt wraps every active move prompt.

| Current shared instruction | Function | Inspection notes |
| --- | --- | --- |
| `Return JSON only. Do not wrap it in Markdown.` | Constrains the provider response format. | Appropriate structural constraint. |
| `Echo targetType, targetId, and selectedMove exactly as provided by the application.` | Prevents the model from changing the app-selected instructional plan. | Appropriate plan-ownership boundary. |
| `Do not expose internal ids, rule ids, rubric labels, scoring fields, or planner metadata in tutorMessage.` | Prevents internal metadata from appearing in learner-facing text. | This prohibits explicit labels and IDs but does not prevent the model from presenting authored misconception wording as if the learner used or understood it. |
| `Use only the authored lesson content and dialogue context supplied in the user message.` | Establishes the content boundary. | Appropriate content constraint. |
| `Follow the selected runtime move policy.` | Assigns pedagogical execution to the move definition. | Keeps shared instructions from competing with move-specific ordering and target-type branches. |
| `Selected move: <moveId>.` followed by `Move prompt: <promptPolicy>.` | Injects the app-selected move definition. | Correctly delegates move-specific behavior to the active registry. |
| The required JSON envelope contains `targetType`, `targetId`, `selectedMove`, and `tutorMessage`. | Enforces the response contract. | Appropriate structural constraint. |

## User-Prompt Inputs Supplied to the Model

| Input section | Supplied value | Inspection notes |
| --- | --- | --- |
| `Problem statement` | The exact authored `opening-tutor-message.value` extracted by the trial-display bridge. | The same required value is supplied to scoring and utterance generation, so reference resolution and tutor wording share the original task context. |
| `Latest student answer` | The learner text extracted from the current SPARC interface event. | The learner's actual response is present. The observed problem is not caused by the response being omitted. |
| `Learner contribution classification` | The current scored contribution classification. | Useful for tone, although the move policies do not specify a complete acknowledgement policy for each contribution type. |
| `App-selected plan` | `targetType`, `targetId`, and `selectedMove`. | Correctly preserves application ownership of target and move selection. |
| `Registered move definition` | Move and output-contract metadata. | Internal structure is available to the model but prohibited from direct learner-facing exposure. |
| `App-selected pedagogical state` | The selected target and move. | Duplicates some plan information for explicit state communication. |
| `Relevant authored target content` | For a misconception, the selected misconception text plus all correct expectations. | This is where authored misconception language can influence the generated wording even when the learner did not use that language. |
| `Current scored planner state` | Expectation scores, misconception scores, selected targets, completion state, and candidates. | Appropriate diagnostic context, but it increases the need for a strict boundary between internal rubric language and learner-facing attribution. |
| `Full dialogue history` | Prior learner and tutor utterances. | Appropriate for avoiding repetition and preserving conversational context. |

| Item | Value |
| --- | --- |
| Learner response | `Well I guess you get $50 every year.` |
| Selected production | `dialogue.scaffold.pump` |
| Selected misconception | `A fixed annual rate means the same dollar amount is added every year.` |
| Generated tutor wording | `That sounds like you’re thinking the account earns the same $50 each year. Can you say a bit more about what “fixed annual rate” means in your example...` |
| Prompt-level explanation | The learner response is present and the misconception selection is semantically reasonable. The generated wording follows the prompt's request for evaluative immediate feedback, then borrows `fixed annual rate` from the authored misconception. No current instruction prohibits treating that rubric terminology as part of the learner's own framing. |

## Prompt-Contract Gaps To Inspect

| Gap | Current behavior | Desired contract question |
| --- | --- | --- |
| Conversational receipt | Every move is told to begin with correctness-polarized immediate feedback. | Should the response first give a short receipt such as “Okay,” “I hear you,” or a faithful paraphrase before any evaluation? |
| Attribution boundary | Authored misconception text is identified as a learner belief to repair. | Should the prompt explicitly forbid claiming or implying that the learner used, knows, or recognizes terminology that appears only in the rubric? |
| Learner-language fidelity | The model receives the learner response but may prefer the more formal authored target wording. | Should acknowledgement and feedback be restricted to the learner's words or a semantically faithful plain-language paraphrase? |
| Feedback calibration | Receipt, correctness feedback, and pedagogical move are combined in each move policy. | Should these become three explicit ordered stages with separate rules? |
| Move-specific feedback | Pump, prompt, hint, and summary share nearly identical acknowledgement language. | Should each move specify when feedback is neutral, positive, partial-credit, or corrective? |
| Misconception repair | The shared prompt prevents endorsement but not learner-facing attribution of rubric language. | Should misconception content be labeled as internal diagnostic context that may guide repair but must not be quoted as the learner's position unless the learner actually expressed it? |
| Regression coverage | Existing tests verify prompt fragments, response schema, and selected-move fidelity. | Should a test cover a learner's plain-language misconception and verify that unrelated rubric terminology is not attributed to the learner? |

## Candidate Prompt Structure For Review

This inspection framework led to the implemented runtime policies recorded above.

| Ordered stage | Proposed responsibility |
| --- | --- |
| 1. Receipt | Briefly acknowledge the learner's actual contribution using the learner's words or a faithful plain-language paraphrase. A receipt may be neutral and need not evaluate correctness. |
| 2. Feedback | Give positive, partial-credit, neutral, or corrective feedback only when justified by explicit latest-response evidence. Do not convert internal rubric wording into something the learner supposedly said or knows. |
| 3. Move execution | Perform exactly the app-selected `pump`, `prompt`, `hint`, `assertion`, `question-deferral`, `question-scope-refusal`, or `summary` move. |
| 4. Attribution boundary | Treat expectation and misconception text as authored instructional context. Never attribute authored terminology or claims to the learner unless supported by the learner's actual response or visible dialogue history. |
| 5. Continuity | Use dialogue history to avoid repeating failed wording and to preserve what the learner has actually established. |
